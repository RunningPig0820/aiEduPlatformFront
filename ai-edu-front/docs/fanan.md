### 1. 图谱 API 契约前置确认

```json
  ]
}
```
- [ ] **0.2** 使用 Swagger / Apifox 等工具 mock 数据，前端先行开发和调试。

### 2. 图谱性能的进一步保障

你提到了后端限制返回节点数（≤50），这很好。但前端仍有优化空间：

- **推荐使用 React Flow 的 `useNodesState` / `useEdgesState` 内置状态管理**，避免不必要的重渲染。
- **布局策略**：初期可让后端返回带有初始位置（如力导向迭代结果）的节点坐标，前端仅做展示；若后端不做布局，前端可引入 `d3-force` 或 `@xyflow/system` 的布局钩子计算初始位置，减少首次渲染时的节点重叠。
- **节点过多时的降级方案**：如果返回 50 个节点仍感卡顿，可增加“简化视图”开关，仅显示核心连接数 Top 10 的节点。

### 3. 树组件改进：虚拟滚动与缓存失效

虽然你决定暂不实现搜索过滤，但 6757 个节点逐级展开后，DOM 节点数仍可能暴增。

- **建议**：在递归树外层包裹一个 `react-window` 或 `react-virtualized` 的 List 虚拟滚动容器。这能显著提升大规模树渲染性能，且实现成本不高（只需将 `<ul>` 替换为 `<FixedSizeList>`）。
- **缓存策略细化**：当前只缓存子节点列表，但未说明失效时机。建议增加：当用户点击“刷新”或重新选择不同教材根节点时，清空缓存，避免陈旧数据。

### 4. 错误处理与边界情况

- **加载失败重试**：在树节点展开和图谱加载失败时，提供“重试”按钮和友好错误提示。
- **无关联图谱时的展示**：目前空状态已规划，可进一步设计一个引导文案：“当前知识点暂无关联图谱数据”。
- **权限拦截**：若用户未登录或无管理端权限，需在 API 层统一拦截并跳转登录。

### 5. 组件间状态联动优化

目前三栏联动通过父组件 props 传递，简单场景够用。但若未来功能扩展（如从详情面板反向定位树节点），建议引入轻量状态管理：

- **方案**：使用 `zustand` 或 `Context + useReducer` 管理选中节点、展开的树节点、图谱数据。可避免 prop drilling，且易于调试。

### 6. 与现有 MPA 项目的集成细节

当前 React SPA 是独立项目，需明确：

- **路由前缀**：若部署在 MPA 同域名下，需处理 `/admin/*` 路由的 Nginx 转发。
- **认证共享**：你已提到 `credentials: include`，需确保后端 CORS 配置允许 SPA 域（若不同端口）。
- **构建输出**：建议 SPA 构建产物放入 MPA 项目 `public/spa/` 目录，通过 Nginx 重写规则挂载。
